3) ESTRUCTURA DEL PROYECTO (COMO EXPLICARLA FACIL)
==================================================

Di esta idea:
"Django organiza el codigo por responsabilidades."

- models.py: define tablas y relaciones.
- forms.py: valida datos de entrada en formularios web.

- reportes.py: exportaciones PDF/Excel/JSON.

Frase de apoyo:

4) MODELO DE DATOS (EXPLICACION SIMPLE)
==================================================

Entidad Cliente:
- Guarda nombre, correo, direccion, telefono.

Entidad Producto:
- Guarda nombre, precio, stock.

Entidad Pedido:
- Pertenece a un cliente.
- Tiene fecha y estado (pendiente, enviado, entregado).

Entidad DetallePedido:
- Une pedido con producto.
- Tiene cantidad y subtotal.
- Subtotal se calcula automaticamente en save.

Relacion clave:
- Un cliente puede tener muchos pedidos.
- Un pedido puede tener muchos detalles.
- Un producto puede estar en muchos detalles.

Regla de negocio importante:
- Si agrego un detalle, bajo stock.
- Si edito un detalle, reajusto stock.
- Si elimino un detalle, devuelvo stock.

6) JWT EXPLICADO PARA PRINCIPIANTE TOTAL
==================================================

Que es JWT en palabras simples:
- Es un texto firmado que identifica al usuario.
- Evita reenviar usuario y clave en cada request.

Que guarda este proyecto en el token:
- user_id
- username
- exp (fecha de vencimiento)

Como funciona en este sistema:

Paso 1: Login
- Usuario envia username y password.
- Si son validos, se crea token firmado con SECRET_KEY.

Paso 2: Guardado
- Se guarda en cookie httponly llamada jwt_token.
- Tambien se puede usar en header Bearer.

Paso 3: Validacion por request
- Middleware lee cookie o header.
- Verifica firma y expiracion.
- Si valido, asigna request.user.

Paso 4: Seguridad basica
- Si token esta vencido o modificado, se rechaza.

Respuesta corta si preguntan "por que JWT":
"Porque me permite autenticar peticiones de forma stateless y reutilizable para web y API."

7) API REST (LO QUE DEBES SABER DECIR)
==================================================

Diferencia web vs API:
- Web devuelve HTML para navegador.
- API devuelve JSON para clientes externos.

Auth API:
- POST /api/auth/login/
- POST /api/auth/registro/
- POST /api/auth/logout/

CRUD REST:
- /api/clientes/
- /api/productos/
- /api/pedidos/
- /api/detalles/

Metodos por recurso:
- GET lista y detalle.
- POST crear.
- PUT actualizar completo.
- PATCH actualizar parcial.
- DELETE eliminar.

Autorizacion:
- Header: Authorization: Bearer TOKEN
- o cookie jwt_token.

Si preguntan "que devuelve":
- JSON con datos de cada recurso.
- codigos HTTP correctos (200, 201, 400, 401, etc).

9) VALIDACIONES Y REGLAS DE NEGOCIO
==================================================

Validaciones en formularios:
- Nombre minimo y formato valido.
- Correo valido.
- Telefono numerico y largo correcto.
- Precio mayor a cero.
- Stock no negativo.
- Fecha no futura en pedido.

Reglas de eliminacion:
- No borrar cliente con pedidos.
- No borrar producto usado en detalles.
- No borrar pedido con detalles.

Control de inventario:
- Crear detalle descuenta stock.
- Editar detalle corrige stock.
- Eliminar detalle devuelve stock.

Frase para defender diseno:
"Priorice integridad de datos para evitar ventas o pedidos inconsistentes."

11) 30 PREGUNTAS TIPICAS DEL PROFESOR + RESPUESTA CORTA
==================================================

1) Que problema resuelve?
Resuelve gestion centralizada de pedidos, clientes y productos con control de stock.

2) Por que Django?
Porque integra ORM, autenticacion, admin y estructura MVC rapida para productividad.

3) Por que MySQL?
Porque es robusto, conocido y suficiente para volumen academico/pyme.

4) Que es un modelo?
Es la representacion de una tabla y su logica basica en Django.

5) Que es una vista?
Es la logica que recibe request, procesa y responde HTML o JSON.

6) Que es una URL en Django?
Es el mapeo de una ruta HTTP hacia una vista.

7) Que hace forms.py?
Valida datos de entrada antes de guardar.

8) Por que hiciste validaciones?
Para evitar datos corruptos y errores de negocio.

9) Que hace DetallePedido?
Relaciona productos con pedidos y calcula subtotal.

10) Donde se calcula subtotal?
En save del modelo DetallePedido.

11) Que hace el middleware JWT?
Toma token por cookie/header, valida y autentica usuario en cada request.

12) Que pasa si token esta vencido?
Se rechaza y el usuario no se considera autenticado.

13) Por que usar cookie httponly?
Reduce riesgo de acceso por scripts del navegador.

14) Que diferencia hay entre sesion y JWT?
Sesion guarda estado en servidor; JWT valida con firma y puede ser stateless.

15) Como probaste API?
Con Postman: login, Bearer token y pruebas CRUD.

16) Que codigos HTTP manejas?
200 OK, 201 Created, 400 Bad Request, 401 Unauthorized.

17) Que es serializer?
Convierte modelo a JSON y valida datos JSON entrantes.

18) Por que separar web y API?
Permite atender navegador y clientes externos sin mezclar respuestas.

19) Como proteges endpoints API?
Con autenticacion JWT y permisos de usuario autenticado.

20) Como manejas stock?
Se descuenta al crear detalle, se ajusta al editar, se devuelve al eliminar.

21) Cual fue reto principal?
Mantener consistencia de stock en todas las operaciones CRUD.

22) Que mejorarias?
Refresh token, roles/permisos, tests automatos y documentacion OpenAPI.

23) Hay paginacion?
Si, en listados web con ListView y paginate_by.

24) Hay reportes?
Si, PDF/Excel/JSON.

25) Que aprendiste realmente?
Modelado relacional, validaciones, JWT, API REST y flujo completo CRUD.

26) Que pasaria con concurrencia alta?
Se debe mejorar con transacciones/locks para stock en alta carga.

27) Que capa modificarias primero para nuevo campo?
Modelo, migracion, serializer/form, vista y template/API.

28) Que significa integridad referencial aqui?
No permitir borrar registros padre cuando hijos dependen de ellos.

29) Cual endpoint de auth usas en API?
POST /api/auth/login/ para obtener token.

30) Como cierras sesion API?
POST /api/auth/logout/ y se borra cookie jwt_token.
