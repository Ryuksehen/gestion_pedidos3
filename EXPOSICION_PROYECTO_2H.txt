3) BLOQUE A - CREACION DE TOKEN (LOGIN)
Archivo: pedidos/views/auth.py
============================================================
Funcion clave: create_jwt_token(user, expires_in=...)

Que hace:
1. Crea header JWT: alg HS256 y typ JWT.
2. Crea payload con:
   - user_id
   - username
   - exp (fecha de expiracion en segundos unix)
3. Firma header.payload con HMAC-SHA256 usando SECRET_KEY.
4. Devuelve token final: header.payload.signature

Lineas importantes a mencionar:
1. Payload incluye exp -> define duracion de sesion.
2. Firma con SECRET_KEY -> evita token falsificado.
3. Todo queda en formato base64url.

En login_view:
1. Si credenciales son correctas, genera token.
2. Lo guarda en cookie httponly llamada jwt_token.
3. Usa max_age basado en settings (duracion centralizada).

Mensaje para profesor:
"No guardo password en token, solo identidad minima y expiracion."


============================================================
4) BLOQUE B - VALIDACION DE TOKEN
Archivos: pedidos/views/auth.py + pedidos/api_auth.py
============================================================
Funcion clave: verify_jwt_token(token)

Que valida:
1. Estructura del token (3 partes).
2. Decodificacion JSON de header y payload.
3. Firma esperada vs firma recibida.
4. exp mayor al tiempo actual.
5. Existencia del usuario en base de datos.

Si todo pasa, retorna user.
Si falla, retorna None.

En API REST:
Archivo: pedidos/api_auth.py
Clase: JwtApiAuthentication

Que hace:
1. Lee token del header Bearer.
2. Si no hay header, intenta cookie jwt_token.
3. Llama verify_jwt_token.
4. Si es invalido, levanta AuthenticationFailed.

Mensaje para profesor:
"La API acepta Bearer y cookie, asi sirve para Postman y para frontend web."


============================================================
5) BLOQUE C - AUTENTICACION EN WEB POR MIDDLEWARE
Archivo: pedidos/middleware.py
============================================================
Clase: JWTAuthenticationMiddleware

Flujo:
1. Si request.user ya esta autenticado por sesion, continua.
2. Si no, busca jwt_token en cookie o Authorization.
3. Si token es valido, asigna request.user = user.
4. Si token invalido:
   - en rutas web privadas -> redirige a login?sesion=seguridad
   - borra cookie jwt_token

Ventaja:
No necesitas usar login de sesion tradicional para mantener usuario activo.

Mensaje para profesor:
"Esto permite autenticacion continua en la web usando JWT, no solo en API."

6) BLOQUE D - ENDPOINTS DE API DE AUTENTICACION
Archivo: pedidos/views/api.py
============================================================
ApiLoginView (POST /api/auth/login/)
1. Valida usuario/clave con serializer.
2. Genera JWT con duracion de settings.
3. Devuelve JSON con token y datos de usuario.
4. Guarda cookie httponly jwt_token.

ApiLogoutView (POST /api/auth/logout/)
1. Limpia estado de auth.
2. Borra cookie jwt_token.

ApiRegistroView (POST /api/auth/registro/)
1. Valida y crea nuevo usuario.
2. Retorna respuesta 201.

Mensaje para profesor:
"Mantengo endpoints separados para login/logout/registro y respuestas HTTP correctas."

7.1) GUIA RAPIDA PARA TESTEAR TIEMPOS (DONDE CAMBIAR)
============================================================
Si quieres hacer pruebas rapidas de expiracion/inactividad, cambia SOLO estos valores:

Archivo principal (obligatorio):
1. proyecto_pedidos/settings.py
   - JWT_ACCESS_TOKEN_LIFETIME_SECONDS
   - JWT_EXPIRY_WARNING_SECONDS
   - JWT_INACTIVITY_TIMEOUT_SECONDS
   - JWT_INACTIVITY_WARNING_SECONDS

Que hacen:
1. JWT_ACCESS_TOKEN_LIFETIME_SECONDS: vida total del token.
2. JWT_EXPIRY_WARNING_SECONDS: cuando empieza el aviso antes de vencer.
3. JWT_INACTIVITY_TIMEOUT_SECONDS: tiempo maximo sin actividad.
4. JWT_INACTIVITY_WARNING_SECONDS: cuenta regresiva final por inactividad.

Archivos que LEEN esos tiempos (normalmente no se cambian):
1. pedidos/views/auth.py
   - login_view: usa JWT_ACCESS_TOKEN_LIFETIME_SECONDS para crear token y cookie.
   - token_status_view: envia warning_seconds, inactivity_seconds e inactivity_warning_seconds al frontend.
   - refresh_token_view: vuelve a usar JWT_ACCESS_TOKEN_LIFETIME_SECONDS al renovar.
2. pedidos/views/api.py
   - ApiLoginView: usa JWT_ACCESS_TOKEN_LIFETIME_SECONDS en login API.
3. pedidos/templates/base.html
   - consulta /api/auth/token-status/ y muestra alertas/contadores.

Pasos de prueba recomendados:
1. Edita settings.py con tiempos cortos (ejemplo: 30, 10, 20, 5).
2. Reinicia runserver (importante para que tome nuevos settings).
3. Cierra sesion e inicia sesion de nuevo (para emitir token con nueva duracion).
4. Espera el aviso de expiracion y prueba el boton renovar token.
5. Deja de mover mouse/teclado para probar inactividad.

Nota importante:
Si en algun momento pones numeros fijos dentro de auth.py o api.py para pruebas,
recuerda volver a dejar lectura desde settings.py para no desconfigurar el sistema.

8) BLOQUE F - AVISO DE EXPIRACION Y RENOVACION
Archivos: pedidos/views/auth.py + pedidos/urls.py + pedidos/templates/base.html
============================================================
Nuevos endpoints web:
1. GET  /api/auth/token-status/  -> token_status_view
2. POST /api/auth/refresh/       -> refresh_token_view

Token status devuelve:
1. expires_in (segundos restantes)
2. warning_seconds
3. inactivity_seconds
4. inactivity_warning_seconds

Refresh token hace:
1. Lee cookie jwt_token.
2. Verifica validez.
3. Si valido, genera token nuevo.
4. Reescribe cookie con nueva expiracion.

Frontend en base.html:
1. Consulta token-status cada 15 segundos.
2. Si token entra en zona de aviso -> SweetAlert con cuenta regresiva.
3. Opciones del usuario:
   - Renovar token
   - Aceptar
   - Salir
4. Si contador llega a 0 -> redirige a login por seguridad.

Mensaje para profesor:
"La expiracion no sorprende al usuario: primero se avisa y se ofrece renovacion."


============================================================
9) BLOQUE G - CIERRE POR INACTIVIDAD
Archivo: pedidos/templates/base.html
============================================================
Logica implementada en JS:
1. Registra actividad del usuario en eventos:
   click, mousemove, keydown, scroll, touchstart.
2. Si no hay actividad por el tiempo configurado,
   muestra alerta de inactividad con cuenta regresiva.
3. Si confirma seguir activo -> reinicia contador.
4. Si no responde o pulsa salir -> logout.

Objetivo:
Evitar sesiones abiertas en equipos desatendidos.

Mensaje para profesor:
"No solo controlo expiracion por tiempo del token, tambien riesgo por abandono de sesion."

11) FLUJO COMPLETO DE SEGURIDAD (RESUMEN PARA DECIR DE MEMORIA)
============================================================
1. Usuario hace login.
2. Se genera JWT firmado con expiracion.
3. Cookie httponly guarda token.
4. Middleware/API validan token en cada request.
5. Frontend monitorea tiempo restante y actividad.
6. Si esta por vencer, alerta + opcion renovar.
7. Si vence o hay inactividad extrema, cierre y redireccion segura.

Frase corta final:
"La seguridad es preventiva, no reactiva: aviso antes de expirar, renuevo controlado y cierre automatico por inactividad."


============================================================
12) PREGUNTAS TIPICAS DEL PROFESOR (RESPUESTAS CORTAS)
============================================================
1) Donde se firma el token?
En create_jwt_token con HMAC-SHA256 + SECRET_KEY.

2) Como verificas que no fue alterado?
Recalculando firma y comparando con compare_digest.

3) Como sabes que sigue vigente?
Con campo exp comparado con time.time().

4) Por que httponly?
Para reducir acceso por JavaScript del navegador.

5) Como renuevas token sin reloguear?
Con endpoint /api/auth/refresh/ que emite nuevo JWT valido.

6) Como informas expiracion al usuario?
Con token-status y alerta con cuenta regresiva en base.html.

7) Como manejas inactividad?
Con timer frontend y eventos de actividad del usuario.

8) Si token es invalido en ruta privada?
Middleware limpia cookie y redirige a login con mensaje de seguridad.
